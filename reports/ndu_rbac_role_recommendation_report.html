Recommendation: approve a layered model instead of adding more boolean flags. For this app, the most practical shape is base workspace role + scope + optional specialist overlays. That keeps the model understandable for demos now and enforceable later without role explosion.

The codebase already covers project, program, portfolio, planning, front-end planning, design, execution, procurement, contracts, vendors, security, SSHER, AI-assisted generation, integrations, billing, and a separate admin domain. A simple admin/member split is too coarse for that surface area.

On the system side, the current single admin role should eventually be split into Platform Owner, Platform Admin, Billing Admin, and Content Admin.

3. Research-Based Design Principles Used

  • Least privilege: users should get only the minimum access needed for their job, both horizontally and vertically.
  • Deny by default: any new resource or endpoint should be blocked until explicitly permitted.
  • Validate on every request: UI hiding is not enough; backend and data-layer checks must also enforce policy.
  • Separate system roles from workspace roles: global administration should not be mixed with day-to-day project management.
  • Use role + scope + relationship: pure RBAC alone becomes brittle once you need object ownership, assigned records, or external sharing.
  • Keep very few owners: high-privilege owner roles should be limited and narrowly scoped.
  • Support viewers and guests explicitly: enterprise tools consistently distinguish editors from read-only and external users.
  • Log permission changes and sensitive access: approval, billing, security, and membership changes should be auditable.

Why that matters here: NDU Project is not only a planning app. It also holds procurement data, contracts, vendor performance, security settings, integrations, and billing/admin operations. That mix makes coarse roles risky.

6. Role Detail Recommendations

Workspace Owner

  • One accountable owner per workspace, with a hard limit of 2-3 total owners for safety.
  • Can assign PMs, Leads, Contributors, Reviewers, and Viewers.
  • Can approve high-risk actions: archive/delete, ownership transfer, final commercial approvals, security overrides.
  • Should have full audit visibility across activity logs and security access logs.

Project / Program Manager

  • Best default role for the person running delivery day to day.
  • Should control planning, execution, coordination, AI drafting, reporting, and member assignment within the workspace.
  • Should not control global admin, coupon/subscription actions, or ownership transfer.

Workstream Lead

  • Use this for procurement lead, security lead, design lead, engineering lead, or ops lead.
  • Give edit rights only for delegated modules and only where they are accountable.
  • Allow module-level approval if you want distributed sign-off without giving full PM or Owner power.

Contributor

  • Best for the majority of internal team members.
  • Can update assigned records and draft content, but cannot approve, reassign roles, or change workspace settings.
  • Useful for engineering, QA, analysts, coordinators, and operations contributors.

Reviewer / Approver

  • Use for governance, sponsors, compliance reviewers, or QA sign-off users.
  • Keep this role read-heavy and approval-focused to preserve separation of duties.

Viewer / Executive

  • Essential for leadership and audit visibility.
  • Should be explicitly read-only, including on AI actions and comments if you want a strict view role.

External Collaborator

  • Do not treat external parties like normal members.
  • Scope them to shared commercial records or status packs only.
  • Default-deny internal planning, security, staffing, and admin data.

7. Recommended Enforcement Shape Later

For this app, enforcement should be built around membership on a workspace object, not only on the user profile. A practical structure would be:

  • users/{uid} for account status and system roles.
  • projects/{projectId}/members/{uid} for project role, module scopes, and overlays.
  • programs/{programId}/members/{uid} and portfolios/{portfolioId}/members/{uid} for higher-level scopes.
  • Optional overlays such as commercial_admin, security_admin, or content_admin.
  • Relationship checks like ownerId, createdById, assigned owner, approver, or explicitly shared external record.
Recommended policy rule: allow an action only when all three conditions are satisfied: (1) role permits it, (2) scope covers the target workspace or module, and (3) any relationship condition is true when required, such as record ownership, assignment, or approved external share.

Minimum controls that should exist once enforcement starts

  • Deny by default for every write path.
  • Validate permissions in UI, service layer, and Firestore rules or trusted backend paths.
  • Log role changes, membership changes, approvals, billing actions, security changes, and denied access attempts.
  • Separate destructive rights from editing rights.
  • Keep a clear distinction between system roles and workspace roles.
